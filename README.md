# FastAPI Example - DDD & Onion Architecture

## 🏗️ 概要

本プロジェクトは、メモ管理（Memo）を題材に、Pythonにおける**ドメイン駆動設計（DDD）**および**オニオンアーキテクチャ**の適用例を示します。

- **関心の分離**: ビジネスルール（Domain）、アプリケーション固有のロジック（UseCase）、技術的な詳細（Infrastructure）、外部インターフェース（Presentation）を明確に分離します。
- **依存性の逆転**: 上位レイヤーが定義したインターフェースを下位レイヤーが実装することで、特定のデータベースやライブラリへの密結合を避けます。

## 🚀 使用している主な技術

- **言語/ランタイム**: Python 3.13+
- **Webフレームワーク**: FastAPI
- **ORM**: SQLAlchemy 2.0 (Async Support)
- **データベース**: PostgreSQL
- **パッケージ管理**: uv
- **マイグレーション**: Alembic
- **ロギング**: structlog
- **バリデーション**: Pydantic v2
- **インフラ**: Docker / Docker Compose

## 🏛️ アーキテクチャの設計思想

ソフトウェアの保守性、テスト容易性、および外部ライブラリへの依存性を最小限に抑えるため、**ドメイン駆動設計（DDD）**と**オニオンアーキテクチャ**を採用しています。

### 採用の意図

- **ビジネスロジックの保護**: ビジネスの核心となるルール（ドメイン）を、Webフレームワークやデータベースといった「実装の詳細」から分離し、システムの中心に据えます。
- **テスト容易性**: 外部システムに依存しない純粋なビジネスロジックを独立させることで、データベースやネットワークがなくても高速かつ確実なユニットテストが可能になります。
- **技術的な柔軟性**: 依存性の逆転原則（DIP）を活用することで、ビジネスロジックに影響を与えることなく、データベース（例：PostgreSQLから他のDBへ）やライブラリの入れ替えが容易になります。

### 依存関係のルール

オニオンアーキテクチャの最も重要なルールは、**「依存性は常に内側（ドメイン層）に向かう」**という原則です。

- 外側のレイヤー（インフラ、プレゼンテーション）は内側を知っています。
- 内側のレイヤー（ドメイン、ユースケース）は外側の詳細（どのフレームワークが使われているかなど）を一切知りません。

## 🧱 各レイヤーの役割と構成要素

### 1. ドメイン層 (`domain/`)

システムの「中核」であり、他の層に依存させずに完全に切り離します。

- **エンティティ (Entity)**: 一意の識別子（ID）を持ち、そのライフサイクルを通じて同一性が維持されるオブジェクトです。
- **値オブジェクト (Value Object)**: 識別子を持たず、属性の内容によって識別される不変（Immutable）なオブジェクトです。ビジネスルールを属性レベルで自己検証（バリデーション）します。
- **リポジトリインターフェース (Repository Interface)**: データの永続化に関する抽象的な定義です。具象的な保存方法は知らず、ドメインが必要とする操作のみを定義します。
- **ドメイン例外 (Domain Exception)**: ビジネスルールに違反した場合に投げられる、ドメイン固有のエラーです。

### 2. ユースケース層 (`usecase/`)

アプリケーション固有のビジネスルールを調整し、ドメインオブジェクトを操作して特定のタスクを実行します。

- **アプリケーションサービス（Application Service）**: 1つのユースケース（操作）を1クラスとして実装し、ドメインオブジェクトを操作します。
- **DTO (Data Transfer Object)**: レイヤー間でデータを転送するためのシンプルなオブジェクトです。インフラ層のデータベースモデルが内側に漏れ出すのを防ぐ役割も果たします。

### 3. インフラストラクチャ層 (`infrastructure/`)

技術的な詳細を提供し、内側のレイヤーが定義したインターフェースを実装します。

- **リポジトリの実装**: ドメイン層で定義されたインターフェースを、特定のデータベース（PostgreSQL / SQLAlchemy）に依存した具象ロジックを記述します。
- **依存性注入 (DI) の設定**: アプリケーション全体で必要なコンポーネントの生成と依存性の注入を管理します。

### 4. プレゼンテーション層 (`presentation/`)

外部世界（HTTPリクエスト）とのインターフェースです。

- **APIハンドラ (FastAPI)**: HTTPリクエストを受け取り、リクエストデータの基本的なバリデーション（型チェックなど）を行い、ユースケース層へ処理を委ねます。
- **スキーマ (Pydantic)**: APIのリクエスト/レスポンスの構造を定義します。これはドメインモデルとは明確に分離され、外部公開用のデータ形式を担います。

---

## 🔄 データの流れ

1. **プレゼンテーション層**: HTTPリクエストを受け取り、Pydanticで型検証します。
2. **ユースケース**: リポジトリのインターフェースを呼び出し、リポジトリ(インフラ層）から**ドメインエンティティ**を取得します。
3. **ドメイン層**: ビジネスロジックが実行され、状態が変更されます。
4. **インフラ層**: ドメインエンティティをDTOに変換し、データベースに保存します。
5. **プレゼンテーション層**: 処理結果をDTO経由でレスポンス形式に変換してクライアントに返します。

## 📁 ディレクトリ構成

バックエンドの構成は以下のレイヤーに基づいています。

```text
backend/app/
├── domain/         # ドメイン層: エンティティ、値オブジェクト、リポジトリの抽象
├── usecase/        # ユースケース層: アプリケーション固有のビジネスルール、操作の調整
├── infrastructure/ # インフラ層: データベースの実装詳細、DI設定
└── presentation/   # プレゼンテーション層: APIエンドポイント、リクエスト/レスポンスの変換
```

## 🛠️ 開発環境の構築方法

### 1. 依存関係のインストール

`uv`を使用して環境を構築します。

```bash
uv sync
```

### 2. アプリケーションの起動

ローカル開発環境を起動します。

```bash
# ホットリロードを有効にしてアプリケーションを起動
docker compose watch
```

起動後、`http://localhost:8000/docs` でSwagger UIを確認できます。

## ✅ バックエンドのテスト

```bash
# ローカル開発中のテスト実行
make test-backend

# または、Dockerコンテナ内で直接テストを実行することも可能です。
make docker-build
make docker-up
docker compose exec backend bash scripts/tests-start.sh -x
```

```bash
# MyPy と Ruff による検証のみ実行
```

## ⚙️ 環境変数

`.env`ファイルを作成し、以下の変数を設定してください。

| 変数名              | 説明                     | 例              |
| :------------------ | :----------------------- | :-------------- |
| `POSTGRES_SERVER`   | データベースのホスト名   | `localhost`     |
| `POSTGRES_PORT`     | ポート番号               | `5432`          |
| `POSTGRES_DB`       | データベース名           | `memo_db`       |
| `POSTGRES_USER`     | データベースのユーザー名 | `postgres`      |
| `POSTGRES_PASSWORD` | データベースのパスワード | `password`      |
| `LOG_LEVEL`         | ログ出力レベル           | `INFO`, `DEBUG` |

## 🛠️ トラブルシューティング

- **データベース接続エラー**: データベースが完全に起動する前にアプリケーションが起動しようとすると失敗します。 `docker compose logs db` でデータベースの状態を確認してください。
- **マイグレーションの不整合**: データベースのスキーマとドメインモデルが一致しない場合、`alembic upgrade head`を再実行するか、マイグレーション履歴を確認してください。
- **環境変数の読み込み失敗**: `.env`ファイルが正しいディレクトリに配置されていること、および`pydantic-settings`などで正しく読み込まれているか確認してください。
